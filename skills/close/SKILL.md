# close

Зафиксировать завершённую работу: задача → `done` + строка в лог сессии проекта. **Закрытие в `done` — только с явного подтверждения пользователя.** Самодостаточно: можно дать любому агенту.

## Главное правило: не закрывай сам

`done` ставит **только пользователь своим «ок»**. Закончил работу — не переводи тикет в `done` молча. Вместо этого:
1. Убедись, что задача существует в статусе `doing` (создай постфактум, если её не было — по контракту `backlog`, но со `status: doing`, не `done`).
2. **Покажи пользователю, что сделано по тикету**, и **попроси подтверждение на закрытие**.
3. Только после явного «ок» — переводи в `done` и пиши в лог сессии.

## Когда вызывать

- Работа по задаче из беклога готова к закрытию → собери отчёт и спроси подтверждение.
- Сделал что-то значимое, для чего задачи не было (фича, фикс, деплой, новый файл/страница/скилл, редизайн, миграция — в т.ч. в инфра/мета-проектах: сам vault, дашборд, скиллы) → заведи задачу постфактум в `doing`, отчитайся, спроси ок. **Не отчитывайся «готово», пока работа не отражена задачей.**

## Шаги

1. **Найти/создать задачу.** Есть — взять. Нет — создать по контракту `backlog` со `status: doing` (RICE проставить).
2. **Верифицировать** (см. ниже) — без этого нечего предъявлять на закрытие.
3. **Отчитаться и запросить ок** по шаблону отчёта (ниже). Дальше — **ждать**. Не переводить в `done`.
4. **После подтверждения:** `status: done`, `updated: <сегодня>`, отметить DoD (`- [x]`); строку в лог сессии карточки проекта (`## Что сделано в сессии <YYYY-MM-DD>` + ссылка `[[tasks/<имя>]]`); обновить роадмап, если закрывался пункт фазы; commit + push.
5. Коротко подтвердить: какая задача закрыта.

Если пользователь говорит «ещё доделай X» — задача остаётся в `doing`, дорабатываешь, снова на подтверждение.

## Шаблон отчёта (перед закрытием)

На каждый закрываемый тикет:

```
**#<номер> — <название>**
— что сделал: <по пунктам, по делу>
— как проверил: <конкретно: собралось, открыл /x в проде → 200, прогнал запрос…; что не проверил — прямо>
```

Несколько тикетов — блок на каждый, затем общий вопрос: «Подтверждаешь закрытие #A, #B?» (можно подтвердить одну, обе или сказать «доделай X»).

## Верификация перед предъявлением (обязательно)

**Закоммитить ≠ сделать.** Прежде чем нести на подтверждение, проверь, что работа реально работает:
- Код — собирается и проходит проверки типов; запущенное — запускается; страница/эндпоинт — открывается; фикс — устраняет проблему на деле.
- Если деплоили — фича жива **в проде** (открыл URL / дёрнул эндпоинт / прогнал запрос).
- Чего проверить **не смог** — **скажи прямо**, не выдавай за проверенное.

В отчёте фиксируй, **как** проверено («собралось + открыл /x в проде → 200»), а не просто «готово».

## Пример

Добавил на дашборд поиск, задачи не было.
→ создать `tasks/task-search.md` со `status: doing`, DoD, RICE;
→ собрать + проверить (открыл /search в проде → 200);
→ ответить: «Сделал по поиску (#154): на /search живой поиск по id/тексту, проверил в проде. DoD выполнен. Подтверждаешь закрытие?»;
→ **ждать ок**;
→ после ока: `status: done`, строка в лог сессии, commit, push.

## Правила

- **`done` — только после явного подтверждения пользователя.** Никогда не закрывать молча.
- Закрытие без записи в лог сессии не считается полным.
- Не удалять задачу при отмене — `status: cancelled`.
- Язык — русский.

## Самопроверка

- [ ] У работы есть задача (в `doing`, если ещё не подтверждена)
- [ ] Работа верифицирована; что не проверил — названо прямо
- [ ] Пользователю предъявлено, что сделано, и запрошено подтверждение
- [ ] `status: done` поставлен **только после явного ока**
- [ ] Строка в логе сессии + (если git) запушено
