# Правила ведения vault

## Назначение
Персистентная память о проектах для LLM-агента и человека. Пользователь не редактирует файлы вручную — все изменения вносит агент автоматически по ходу разговора.

---

## Когда читать

- В начале любого разговора, связанного с проектами — соответствующий `projects/<проект>/<проект>.md`
- Если тема разговора неясна — `_dashboard.md` для общей картины
- Перед созданием задачи — текущий беклог проекта

## Когда обновлять

Обновлять **сразу**, не дожидаясь конца разговора, когда появляется:
- Новый статус проекта или задачи
- Новая задача или идея
- Принятое решение по архитектуре / подходу
- Новая метрика или изменение цели
- Информация о ЦА, видении, проблеме

---

## Карточка проекта (`projects/X/X.md`)

- `status`, `priority`, `updated` в frontmatter
- Разделы: Суть, ЦА и боль, Видение, Метрики, лог сессий («Что сделано в сессии <дата>»), Связанное
- Топ-задача: ссылка `[[tasks/имя]]` на задачу с наибольшим RICE

## Задача (`projects/X/tasks/имя.md`)

- Создавать из `_templates/task.md`; имя файла — короткое, snake-case/kebab-case, на английском
- **`id` — сквозной номер тикета**: максимальный `id` среди всех `projects/*/tasks/*.md` + 1. Не переиспользовать и не пропускать
- **Оценка времени**: `est_days` в идеальных днях (1 день ≈ 6 фокус-часов); большинство задач 0.5–3
- **RICE**: `rice_reach` (1–10), `rice_impact` (1–5), `rice_confidence` (50–100), `rice_effort` = `est_days / 5` (чел-недели, min 0.1). RICE = (reach × impact × confidence%) / effort
- Статусы: `todo → doing → done`; ещё `blocked` и `cancelled`
- Для багов/инцидентов тег `bug`: `tags: [task, bug]`
- Разделы:
  - **Что нужно сделать** — чёткое описание
  - **Почему важно** — связь с целью проекта
  - **Критерии готовности (DoD)** — чеклист наблюдаемых условий результата: что должно стать правдой (поведение/результат), НЕ действия. Главный раздел задачи; отмеченные галочки = прогресс
  - **Пререквизиты** — зависимости и блокеры (или «нет»)
  - **Вопросы** — что прояснить до старта; писать в файл, не спрашивать в чате
  - **Заметки** — происхождение, контекст, ссылки; сюда же план реализации, если нужен (отдельной секции «План» нет)

## Когда «В работе» пусто

Если ни одной задачи в статусе `doing` нет, предложи пользователю **3 задачи, с которых начать** (навык `sync`):
- **Проектов с открытыми (`todo`) задачами меньше 3** → топ-3 открытых задач по убыванию RICE.
- **Проектов 3 и больше** → по одной задаче из трёх разных проектов: у каждого проекта берётся самая дорогая открытая задача, затем три проекта с наибольшим RICE топ-задачи. Так фокус распределяется по проектам.

---

## Индекс задач (`projects/X/tasks/tasks.md`)

Dataview-таблица всех задач проекта (см. пример в `projects/example-project/tasks/tasks.md`). Не редактируется вручную — строится из frontmatter. RICE для `done`/`cancelled` показывается как «—».

## Роадмап (`projects/X/roadmap.md`)

Фазы с чекбоксами. Обновлять по мере прояснения плана: выполненное отмечать, новое добавлять.

---

## Инбокс (`_inbox/`)

Сырые мысли «на разбор» — по одному markdown-файлу на запись:

```
---
created: <ISO-дата>
project: <slug или отсутствует>
tags: [inbox]
---

<текст мысли>
```

Цикл разбора (навык `inbox`): запись → полноценная задача с RICE → файл записи удаляется → commit+push. Инбокс не остаётся наполовину разобранным.

---

## Как создать новый проект

1. Папка `projects/<slug>/`
2. `projects/<slug>/<slug>.md` из `_templates/project.md`, тег `project` в frontmatter
3. `projects/<slug>/roadmap.md`
4. Папка `tasks/` + `tasks.md` (индекс, скопировать из примера и поправить путь в FROM)

---

## Отслеживаемость действий

Каждое значимое действие агента отражено в задачах:
- Агент пишет/меняет код → задача существует (статус `doing` или `done`)
- Фича оказалась уже реализованной → задача создаётся со статусом `done`, роадмап обновляется
- Фича обсуждается как будущая → задача `todo`

**Гейт (главное правило): прежде чем отчитаться пользователю о выполненной работе по проекту — убедись, что работа отражена задачей.** Нет задачи — заведи постфактум сразу со `status: done` (навык `close`). Проверяй на каждом ходе, где что-то сделал, а не «в конце сессии».

- **Что считается работой, требующей задачи:** новый код/фича, фикс бага, деплой, новый файл/страница/скилл/компонент, редизайн, миграция, правка конфигурации.
- **Не требует задачи:** чтение, исправление опечатки, ответ на вопрос без изменений в репозитории.
- **Касается и инфра/мета-проектов** (сам vault, дашборд, скиллы, шаблоны) — не только «продуктовых». Частый перекос: продукту задачи заводятся, работе над инструментами — забывается. Считается одинаково.
- Завершил задачу → закрывай навыком `close`. **`done` — только после явного подтверждения пользователя.** Закончил работу → задача в `doing`, предъяви что сделано по тикету по шаблону отчёта (**#номер — название · что сделал · как проверил**) и попроси ок; не закрывай сам. После ока — `status: done` + строка в лог сессии карточки проекта.
- **Закоммитить ≠ сделать.** Прежде чем нести на подтверждение, проверь, что работа реально работает (собралось, запустилось, страница открылась, в проде живо). Что проверить не смог — назови прямо, не выдавай за проверенное. В DoD и отчёте фиксируй, *как* проверено.

Цель: по vault всегда можно восстановить, что делалось и что сделано. Если канбан проекта пуст, а работа по нему велась — это сломанная отслеживаемость, а не «нет задач».

## Git

Если vault — git-репозиторий: после содержательных правок агент делает `git add -A && git commit && git push`. Перед разбором инбокса — `git pull --rebase --autostash` (записи могут приходить с других устройств).

---

## Чего не делать

- Не удалять информацию без явного указания пользователя
- Не менять структуру папок и заголовков файлов
- Не задавать в чате вопросы по задачам — писать их в раздел «Вопросы»
